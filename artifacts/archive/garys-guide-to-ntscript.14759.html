As it stands, nobody does Signal Tech competently. The current meta for how people play Signal Tech is analogous to a world where knowing to max out the pressure on the Waste In pump in Atmos makes you a grandmaster Atmos Tech.

Why would I say that? Because communication is the biggest part of this game, and being signal tech makes you god over all communication that occurs on the station. Think about it; you can control exactly what is said, who says it, how they say it, and when they say it, from anywhere in comms range. Being able to handle and manipulate information effectively can make or break rounds.

Unfortunately all this shit is hidden behind the wall that is NTscript, and since there's not too much real documentation on it, nobody really knows what to do with it. Luckily, I can rectify those problems.

Like Korea, this guide is split into two parts:
  1. How to fuckin' write NTscript, taught as if you have never programmed a damned thing before this
  2. Unique GaryOS - sponsored tricks to give your scripts actual practical purpose.
So, lets get started.


HOW TO FUCKING WRITE NTSCRIPT

Alright, first, lets explain how to get to the point of being able to start writing anything.

Once you wonder on over to your office, the Telecomms Control Room, you are met with four beepboopers:

Here's what they are:
  • The Message Monitor lets you read everyone's saucy PDA messages, and send your own under fake names. We're not talkin' about it at all.
  • The Server Monitor lets you get the logs of every server on a given network, including error logs. We'll talk about those later.
  • The Traffic Control Monitor is the computer for scripting. This is the only one you truly require to do your job correctly.
  • The Automated Announcement System broadcasts messages (processed by the common server) about new people arriving. Clicking on it lets you set what the messages are. Useful for memes, warning people of immediate danger on-arrival, and testing comms to make sure it's working.
Going to the Traffic Control one, putting your ID into it (by clicking on the NO ID "link") and clicking on the [Load Buffer] button will connect you to every server on the tcommsat network, like so:

So, each server has its own independent NTscript environment. The Common Server can run a completely different script with completely different stored variables to every other server. Clicking on a server chooses which server, and thus which department, you will be annoying with your shitty memescripts today.

After choosing a server, you get this window:

So, lets explain what each thing does.
  • [View System Log] Lets you see if someone else has been snooping around this computer. If a "System Administator" was fucking with it, you know it was either the AI or a cyborg.
  • Signal Execution decides whether your script is on or off.
  • [Edit Code] is how you get to the IDE, which is the window behind the window I'm talking about.
The IDE is how you will actually be doing things. Here it is here:

Here's what each button bla bla you get the idea:
  • Save keeps whatever you have written into the IDE saved for when you come back. Mostly useless, since you should be using an outside text editor to write code.
  • Compile will attempt to read through, process, and save your code onto the server for use. It will either pop out a Compilation Successful! message or a list of Compile Errors (as shown). This must be done before the server can do anything with the code you've written.
  • Execute is supposed to run your script once and only once. You'll never need to really do that.
  • Revert is some sort of shitty Undo that doesn't really work. Again, use another text editor.
  • Clear Memory erases your script and any mem()orized variables on the server.
In middle, there's a big ol' text box where you put your text. THAT is where you write code.
---------------------------------------------------------------------------------------------------------------

Okay, now, lets learn how this actually works.

But, first, I want to explain:
You should not be really reading this guide in one go, if you're brand new to programming. I would strongly recommend, after learning this or that big concept, you go into the IDE yourself and try to program something using that concept. This is a skill that is best learned through self-experimentation and fucking around, rather than just doing and remembering what a guide says. The reason this guide exists is to merely give you a helping hand and solid starting point to work from, not to make you a master by the end of it. Also, I recommend having this page with you at all times. Even I keep it in a tab at all times, in case I forget how some function takes input or what frequency Security is on.

Whenever a message ends up on the server for the script to work on, it gets these variables that are defined before the script even starts. They store things like who said this message, what the message is, etc. At the end of the script, the server takes a few variables that it needs for broadcasting and then uses those to send out the final, script-changed message.

Lets use an example. Lets say my name is Gary Lafortune and I am the Captain, and I say this:
Gary Lafortune says, ";Liking clowns isn't gay."

Here's the resulting variables that are declared for your script's use:
$source = "Gary Lafortune"
$job = "Captain"
$say = "says"
$content = "Liking clowns isn't gay."
$freq = 1459 (This being the frequency the common channel is on)
$language = HUMAN
$uuid = "Gary Lafortune" (This is a lot more important than it looks; we'll talk about it later)

And when the server sends it back out, it reads, by default:
[COMMON] Gary Lafortune says, "Liking clowns isn't gay."

Which, to the server and to your script, reads, more or less:
[COMMON] $source $say, "$content"

Changing the value stored in $source, $say, or $content affects the final message transmitted.
(The exact verb-variable in the middle depends on what $content is. If it's a question, it will be $ask instead. If there's one exclamation point at the end, it uses $exclaim. $yell is used when there are two or more exclamation points at the end.)

Will turn the previous message into this:
[COMMON] Gay Guy says, "Liking clowns isn't gay."

Lemme explain that whole line real quick:
  • The whole phrase "$source = " means "Set the value of $source to be..." or, as I read it, "$source set to ..." It does not mean the question "Hey, does $source equal ...?" I'll explain later.
  • The reason there's quotes around "Gay Guy" in the script is because it's a string, which is something that holds a bunch of characters, a string of text. If I didn't have quotes around it, it would think I'm trying to do something else.
  • Keep in mind that every discrete line in your code should be ending in a semicolon. That is how the compiler knows when a specific command ends.

Now, two more examples before I let you free. Lets say you want your job to be a random meme each time you talk, and only you get to have this random meme job.

First, lets create the thing the checks that it's really you. Here we'll need an If Statement, which is a thing that lets you do conditional shit like this. Here's what it looks like:

Lemme explain what's up:
  • The If-Statement looks at the thing in parentheses to its right and sees if it doesn't evaluate to FALSE. If it doesn't, then it runs the code in braces below it. So, as long as the thing inside eventually becomes anything that isn't FALSE, then the code block below it will run.
  • Here inside the If-Statement we have the phrase " $source == "Gary Lafortune" ", which asks the question "Is $source equal to the string "Gary Lafortune"?" If it does, that phrase transforms into the value TRUE. If not, it transforms into the value FALSE.
  • Inside my if-statement block there's this green text. That isn't >le 4chan meme, those are comments, which let you write notes into your code itself that aren't looked at by the compiler. They literally do nothing but let you remind yourself (and others) how some piece of spaghetti you wrote works. You can also comment-out pieces of code, so that you can keep them for later while still not executing them, which is helpful for debugging.

Alright, one final example involving mem() and commands and you're on your own.

Lets say you want to be able to disable your memejob thing on the fly, without having to manually remove it from the script. Think about what the code has to end up being. You gotta have a command that you can run to change a variable that tells the script whether it should be memeing or not.
But the problem is that once a script is done running and the message is sent, all variables declared are deleted. The only things that remain afterwards include the script itself and variables stored into memory via mem().
mem() itself does one of two things:
  1. If you give it a string, i.e. mem("memes"), it returns the value of the mem()orized variable called memes.
  2. If you give it a string and then a thing, like mem("memes",TRUE), it will store the thing into that variable, to be kept for later.

  • That ! thing does one of two things: If the value given isn't FALSE, it makes it FALSE. If it is FALSE, it makes it TRUE. It's a flippity-flopity.
  • That && thing (it means AND) lets you test for TWO things to be true at once. Here, it tests that both memes are enabled AND that the person talking is a friend.

A boot() function is necessary if you want your mem()orized variables to start with a value that isn't FALSE.

Now, some words of wisdom here: This shit means that you can generalize pretty much everything you're doing into singular, simple things. Anything that you do a lot that varies based on just a few inputs, you can stuff into a function like this, and just call that function whenever you need to do that thing, no matter how complex that thing is. Not only does this save you typing time, but it also minimizes the conceptual complexity of your program, making it easier to understand and fix when there's a problem. My ol' pappy once told me, "In programming, son, you should never do anything more than twice, and twice is pushing it." Learning to do this is necessary in order to produce clean, functional code.

"But Gary," I hear you say, "That doesn't really explain EVERYTHING about how NTscript works! How do I do [thing???]"

If you scroll up and see how I started this Part I, I noted that this was mostly supposed to be a guiding hand helping you discover and get used to shit on your own. All this is here for is to give you an indication as to how to write shit and what to do.

But, yeah, you are going to need to see more examples to actually git gud, so maybe I should list some things you can do via NTscript, like so:


CRITICALLY IMPORTANT SHIT TO PUT IN YOUR SCRIPT

$uuid & Fucking With The AI
Remember $uuid? It was right at the start of this guide, and I said I would talk about it later. It is now later.
$uuid is the person that the AI is linked to whenever they see you talk in the chatter, and it is normally the exact same as $source. If they click that link, they are sent to the person of the relevant $uuid.

So, imagine having to go through the entire list of crewmen every time you want to find anybody as an AI. That is what you can force upon the AI if you set the $uuid to an empty string, or if you set it to some other random person's $uuid, which can be crucial if the AI is rogue. This has a fair bit of creative potential, so I won't provide any examples for this one.

A Note About Malicious Code
Most folk would assume that your antag code would include that AI-tracking fuckery I was talking about earlier, and maybe a function to mute shit or what have you. While these are things to have in your pocket, you must understand something very important about antagscripts:

A good antagscript must make communication confusing enough that nobody can get any information across, but not confusing enough that they realize what's up and start using alternative methods of communication.

For example, a thing that randomly swaps letters in a dyslexic pattern, or a thing that only sometimes turns the content into "...", or a thing that activates pirate speech randomly. Small things like that, en masse, can and should slow down comms to a crawl, and, if done right, nobody should expect that you're doing anything with malicious intent. This is the philosophy you should have when you're designing your antagscripts / commands.

def abbrev($name) // Abbreviates the given 'name'
{
    $fulljobs = vector("Secbot","Medbot","AI","Clownbot",
    "Engibot","Cyborg","Clown","Peacebot","Unknown");
    $fullnams = vector("chaos magicarp","Gary Lafortune","Jeanna Lafortune","Joe Hobostein");
    // ^ Jobs & names that imply printing the full name no matter what
    if(!find($fulljobs,$job) && !find($fullnams,$name))
    {
        $namevect = explode($name," ");
        $namelen = length($namevect); // Actually how many words are in the name
        if(length($namevect) < 3)
        {
            $f = at($namevect,1);
            $bannedfirst = vector("The","Mr.","Dr.","Syndicate","NTP");
            $l = at($namevect,2);
            if(!find($bannedfirst,$name) && (length($l) > 3))
            {
                $last = substr($l,1,2);
                return $f + " " + $last + ".";
            }
        }
    }
    return $name;
}
